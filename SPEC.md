# AGH 多子系统 Demo 产品规格

## 目标

本 Demo 用于在正式开发前与客户确认完整业务流程、角色职责、页面结构和系统边界。它不是单页售前展示，而是一套可操作的多角色产品原型。

## 系统边界

系统提供一个仅用于选择入口的 Portal 页面，不展示业务数据、不切换角色。六个子系统分别从独立登录页进入，并拥有自己的导航、首页和业务页面：

1. 患者端：咨询、资料、方案、行程、治疗进度和随访。
2. 马来服务端：线索 CRM、签约、资料完整性、Case 提交和本地协调。
3. 中国运营端：病历翻译整理、专家分配、医院匹配和跨国交接。
4. 专家端：病例阅读、MDT、治疗建议和复查解读。
5. 医院承接端：接诊、床位、检查、治疗、费用和出院资料。
6. 归国健康管理端：五阶段计划、随访、用药、检验、康复、护理、应急和质控。

## 主线场景

`咨询建档 → 资料签约 → 中国运营整理 → 专家评审 → 医院承接 → 在华治疗 → 出院归国 → 五阶段健康管理 → 异常复评`

关键操作必须同时更新患者阶段、任务队列和审计时间线。

## 案例与数据

- 主案例：林秀英，马来西亚肺腺癌患者，贯穿完整流程。
- 辅助案例：新咨询、在华治疗、高危随访各一例。
- 所有数据存入 Pinia，并持久化到 `localStorage`。
- 评审、治疗、费用、行程、随访、授权、备注和附件按 `cases[caseId]` 隔离。
- 列表与统计由统一状态实时派生，不维护独立患者静态副本。
- 各系统会话相互独立，共享同一套业务种子数据。
- 重置能力保留在共享状态层，便于演示前恢复基线。

## 页面与交互要求

- 每个内部系统至少包含工作台、业务列表、业务详情/处理页和跨端任务页。
- 患者端至少包含首页、医疗资料、治疗方案、赴华行程和归国随访。
- 页面主按钮必须产生明确反馈；跨端动作必须生成接收方待办或改变接收方可见状态。
- Portal 点击系统后在新标签页打开对应登录入口。
- 每个菜单页面必须匹配实际业务场景，至少包含业务字段、筛选/导航、记录展示和处理动作；记录应能进入详情或办理页面。
- Demo 不连接真实后台，但页面层级、表单、列表、详情、状态反馈和跨页面跳转应与正式系统使用方式一致。
- 患者首页展示平台价值、选择理由、合作医院、服务流程和联系方式；个人医疗服务从底部菜单进入。
- 患者资料、方案、行程、随访及详情必须使用移动端专用页面，不复用桌面工作台。
- 内部系统集合页展示全体患者或业务记录，不默认固定某一患者；患者信息仅在明确的病例详情和办理页面展示。
- 列表筛选项必须改变实际记录集合；详情页必须展示用户点击的当前记录及其对象类型，不能回退到默认患者。
- 跨国协同任务的开始处理、继续处理和完成操作必须有可见状态变化和处理表单。
- 分配专家后专家端出现评审任务；发送医院承接申请后医院端出现接诊任务；完成出院交接后健康管理端出现建档任务；升级异常后专家端出现复评任务。
- 未登录访问业务路由时，必须跳转到该系统自己的登录页。

## 归国五阶段

1. 归国前准备（治疗结束前 3 天）
2. 归国适应期（1–7 天）
3. 短期康复期（1–3 个月）
4. 中期稳定期（3 个月–1 年）
5. 长期随访期（1 年以上）

每阶段包含动态频次、责任角色、复查项目、用药、康复和异常响应，不采用固定四节点模型。

## 非功能要求

- 全系统中英文切换。
- 患者端按移动设备设计，内部工作台按桌面设计。
- 关键医疗数据显示授权和版本概念。
- 所有高危事件必须可升级、分派和留痕。
- 不连接真实医疗、支付、WhatsApp、AI、HIS/EMR/LIS/PACS 接口。

## 主流程准入规则

1. 患者资料完整度达到 100% 且跨境授权有效后，马来服务端才能提交中国运营。
2. 专家正式意见签署完成后，中国运营才能发送医院承接申请。
3. 医院确认承接、患者确认费用、患者和医院确认行程后，才能完成赴华交接。
4. 患者进入术后或治疗完成状态，且双语出院资料、影像病理、带药、复查计划和患者签收全部完成后，才能完成出院交接。
5. 医院完成出院交接后，健康管理端才能生成五阶段计划。
6. 高危预警升级必须填写风险评估，关闭必须填写处置结论。

## 需求确认

客户评审以 `REQUIREMENTS_MATRIX.md` 为准。任何字段、状态、准入规则、责任角色或权限边界发生变化时，应先更新矩阵和决策记录，再调整页面实现。

## 商业范围治理

- 客户三期报价范围以 `参考资料/跨境肿瘤医疗协同平台建设方案.pptx` 为唯一依据。
- `SCOPE_MATRIX.md` 维护每项能力的所属期次、A/B/C 分类、深度上限和增购清单。
- A 类按对应期次实施；B 类不得超过已定义深度；C 类不得开发，除非取得书面变更和新增报价。
- 宣传材料、医疗 SOP 和业务参考文档不得作为新增软件交付范围的依据。
