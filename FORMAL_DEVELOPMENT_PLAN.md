# AGH 正式开发推进蓝图

本文件用于把 Demo 阶段确认出的业务原型，转换为正式开发阶段的工程化交付思路。后续讨论正式开发、报价、排期和团队协作时，以本文件、`SCOPE_MATRIX.md`、`REQUIREMENTS_MATRIX.md`、`DECISIONS.md` 为持续更新的事实来源。

## 1. 总原则

- Demo 是需求确认原型，正式系统不能直接在 Demo 代码上无限扩展。
- 正式开发先冻结范围和验收口径，再设计架构和数据模型。
- 正式开发按“分期价值闭环”推进，不在一期铺开全链路，避免范围发散和交付失控。
- 先解决最痛的患者端与马来团队运营端，再接入国内诊疗协同，最后做 AI 和高附加值能力。
- 先状态机、权限和审计，再做高级体验。
- 先人工闭环，再引入 AI、BI、WhatsApp 等自动化能力。
- A 类按期交付，B 类限深交付，C 类进入增购清单，不因开发便利自然扩展。

## 2. 阶段一：冻结范围和验收口径

正式开发启动前，需要先确认：

- 一期、二期、三期分别交付什么。
- 每个功能属于 A/B/C 哪类范围。
- 每个角色能看什么、能做什么、不能做什么。
- 每个业务状态如何流转，哪些动作允许或禁止。
- 每个核心场景需要哪些字段和校验规则。
- 每个功能如何验收，如何证明完成。

产物：

- `SCOPE_MATRIX.md`
- `REQUIREMENTS_MATRIX.md`
- 字段字典
- 状态字典 / 状态机说明
- 权限矩阵
- 验收用例清单

## 3. 阶段二：系统架构和数据模型设计

正式系统需要从 Demo 的 `localStorage` 原型转为真实工程架构：

- 前端 Web / H5
- 后端 API
- 数据库
- 文件存储
- 认证与权限
- 审计日志
- 消息通知
- 部署、日志、监控和备份
- 后续 AI / BI / 第三方消息接口

核心数据模型建议：

- Patient
- Case
- Document
- Consent
- Task
- Review
- HospitalMatching
- Treatment
- Billing
- Travel
- Discharge
- Followup
- Alert
- Notification
- AuditEvent
- Organization / User / Role

关键设计原则：

- Case 是主业务聚合根。
- 资料、授权、评审、医院、费用、治疗、随访都围绕 Case 组织。
- 列表和统计从业务状态派生，避免维护多套重复数据。
- 关键动作必须落库、触发状态变化、生成审计事件，并在必要时生成通知或任务。

## 4. 阶段三：按分期价值闭环推进

第一版正式系统不应先铺满所有菜单，也不应一上来打通从马来到国内医院再到归国健康管理的完整链路。正式开发应优先做“最痛、最短、最容易看到价值”的一期闭环：

```text
患者端
↔ 马来团队运营端
```

一期目标是让患者和马来团队先真正用起来，解决咨询、报名、资料、授权、跟进、Case 准备、通知和运营管理问题。国内诊疗协同、医院承接、健康管理和 AI 能力后移到二期、三期。

### 一期闭环：患者端 + 马来团队运营端

一期只开发患者端和马来团队运营端，形成海外运营数字化闭环：

```text
患者咨询 / 报名
→ 马来团队建档和跟进
→ 患者上传资料
→ 患者签署授权
→ 马来团队整理资料和合同
→ 马来团队维护 Case 准备状态
→ 患者查看进度、通知和服务安排
→ 马来团队获得运营统计和待办提醒
```

一期不做国内运营、专家、医院、健康管理和 AI 的正式生产能力；这些可以在一期中以占位状态、外部备注或人工线下流程承接，避免系统范围发散。

一期必须真实连接：

- 患者账号与马来团队账号
- 患者资料与 Case 准备状态
- 文件上传、版本和授权
- 马来团队跟进记录、任务和通知
- 患者可见进度和提醒
- 基础权限与审计

### 二期闭环：国内诊疗协同接入

二期在一期稳定使用的基础上，接入国内几个板块，同时继续迭代患者端和马来端：

```text
一期已准备 Case
→ 中国运营接收
→ 分配专家
→ 专家评审
→ 医院匹配
→ 医院承接
→ 费用/行程确认
→ 出院交接
```

二期重点是国内诊疗协同，而不是推倒一期重来。患者端和马来端在二期需要同步升级，例如展示专家意见、医院方案、赴华行程、费用确认和交接状态。

### 三期闭环：AI 与高附加值能力

三期才进入 AI 和高附加值能力：

- AI OCR
- AI 病历摘要和术语翻译
- 智能随访与预警建议
- BI 患者、渠道和收入分析
- WhatsApp 或第三方消息增强
- 医学知识库问答

三期能力必须建立在一、二期已经沉淀的数据、流程、权限和审计之上，不能提前变成“为了 AI 而 AI”的演示功能。

## 5. 阶段四：按期次扩展模块

每一期都要有自己的价值闭环，同时允许前一期在后一期间继续迭代。

### 一期：海外运营数字化

只做患者端和马来团队运营端。优先能力：

- 患者门户 H5
- 马来 CRM
- 咨询 / 报名
- 患者档案
- 患者跟进
- 资料上传
- 授权管理
- 合同 / 签约状态
- Case 准备状态
- 基础任务协同
- 基础通知
- 角色权限
- 基础运营统计

一期明确不做：

- 中国运营正式工作台
- 专家评审正式流程
- 医院承接正式流程
- 归国健康管理正式流程
- AI / BI / WhatsApp 自动化

### 二期：中国诊疗协同

接入国内板块，并持续升级患者端和马来端。优先能力：

- 中国运营接收
- 诊疗资料中心
- 翻译 / 医学核验
- 专家评审
- MDT
- 医院匹配
- 医院承接
- 费用预估
- 赴华交接
- 出院交接
- 患者端展示专家意见、医院方案、费用、行程和交接状态
- 马来端查看国内协同进度、补资料任务和患者沟通状态

### 三期：智能运营升级

在一、二期数据和流程稳定后，再做 AI 和高附加值能力。优先能力：

- 随访计划
- 预警闭环
- BI 分析
- AI OCR / 摘要 / 翻译
- WhatsApp 或第三方消息
- 医学知识库问答

注意：AI、WhatsApp、Token、云服务和第三方费用按报价规则另行确认，不自动纳入基础交付成本。

## 6. 阶段五：质量体系

每个功能完成标准不是“页面做完”，而是：

```text
功能实现
+ 权限正确
+ 状态正确
+ 数据落库
+ 审计留痕
+ 错误路径明确
+ 测试通过
+ 文档更新
```

正式开发需要建立：

- 单元测试：状态机、权限、费用、资料版本。
- API 测试：接口输入输出、错误码和边界条件。
- 集成测试：跨角色任务流转。
- E2E 测试：主链路端到端跑通。
- 权限测试：不同角色访问边界。
- 审计测试：关键动作必须留痕。
- 文件测试：版本、授权、访问记录。
- 回归测试：每次发布前自动执行。

## 7. 阶段六：部署、运维和安全

正式系统必须提前设计：

- HTTPS 和域名
- 生产 / 测试 / 演示环境隔离
- 数据库迁移
- 文件存储和备份
- 日志和监控
- 故障回滚
- 权限审计
- 医疗数据脱敏
- 数据授权、撤回和访问记录
- 跨境访问和数据边界策略

如果系统处理真实患者数据，医疗数据安全、跨境授权、访问审计和隐私合规必须作为一等需求处理，不能后补。

## 8. 推荐推进节奏

```text
第 0 步：把 Demo 归档为需求原型
第 1 步：整理正式开发蓝图
第 2 步：确认一期患者端 + 马来团队运营端范围
第 3 步：设计一期数据模型、权限、文件和通知
第 4 步：搭建正式工程骨架
第 5 步：交付一期海外运营闭环
第 6 步：客户试用、收集反馈并迭代一期
第 7 步：进入二期国内诊疗协同，同时升级一期端
第 8 步：进入三期 AI / BI / 高附加值能力
```

## 9. 工作方式建议

- Demo 优化阶段可以直推 `main`，保持快速迭代。
- 正式开发阶段建议改为功能分支 + PR + 审查 + 合并。
- 每个正式功能进入开发前，先确认范围分类、字段、状态、权限和验收用例。
- 每个阶段交付前，必须给出测试、构建、部署和风险说明。

## 10. 当前后续讨论入口

下一次讨论正式开发时，优先从以下问题开始：

1. 一期患者端和马来团队运营端的最小可上线闭环具体包含哪些页面？
2. 数据库和后端技术栈如何选择？
3. 一期文件存储、授权和访问日志是否作为硬性范围？
4. 一期是否需要保留中国运营/专家/医院/健康管理的占位状态或完全不做？
5. 二期接入国内板块时，患者端和马来端需要同步升级哪些能力？
6. 客户评审签字前，字段字典和接口契约需要做到多细？
7. 正式开发团队是单体应用推进，还是前后端/API/测试分工推进？
